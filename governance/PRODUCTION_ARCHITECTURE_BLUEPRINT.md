# URAI-Jobs: Production Architecture Blueprint

## 1. Overview

URAI-Jobs is a comprehensive job board, applicant tracking, and recruitment platform built on a serverless architecture using the Google Cloud Firebase ecosystem. The system is designed for multi-tenancy, ensuring strict data isolation between different organizations, and prioritizes security, scalability, and maintainability.

## 2. Technology Stack

- **Frontend**: A Single Page Application (SPA) built with **Vite, React, and TypeScript**. This provides a fast, modern, and type-safe user experience.
- **Backend**: Serverless functions written in **TypeScript** and running on **Node.js 20** in the Firebase Cloud Functions environment.
- **Database**: **Cloud Firestore** in Native Mode, utilizing a multi-tenant data model for data isolation.
- **Storage**: **Google Cloud Storage** for securely handling sensitive PII data, specifically applicant resumes.
- **Authentication**: **Firebase Authentication** for user identity management, primarily for the protected admin console.
- **Hosting**: Static assets are served globally via **Firebase Hosting**, with CDN and automatic SSL.

## 3. System Architecture

### 3.1. Frontend Application

- The frontend is a pure client-side application built as an SPA.
- It is hosted on Firebase Hosting, with all routes rewriting to `/index.html` to enable client-side routing.
- The build output from `/web` is placed into the `/public` directory, which is the root for Firebase Hosting.

### 3.2. Backend Services (Cloud Functions)

The backend logic is composed of discrete, event-driven Cloud Functions:

- **Firestore Triggers**: These functions react to changes in the database to perform actions such as creating public job projections or updating statistics. All triggers are built with multi-tenancy as a primary concern.
- **Callable Functions**: For client-invoked actions that require administrative privileges or secure operations, such as changing an application's status or generating secure upload URLs for resumes.
- **HTTP Functions**: Standard HTTP endpoints for tasks like health checks.
- **Scheduled Functions**: Cron-job-like functions for periodic tasks, such as generating daily digest reports.

### 3.3. Data Model (Firestore)

The Firestore database employs a multi-tenant model. All organization-specific data is nested under a top-level `orgs` collection, keyed by a unique `orgId`.

- `/orgs/{orgId}/jobs/{jobId}`: Private job data, only accessible to members of the organization.
- `/orgs/{orgId}/jobPublic/{jobId}`: A public-safe projection of open jobs, used for the public job board.
- `/orgs/{orgId}/applicants/{applicantId}`: Applicant profiles specific to an organization.
- `/orgs/{orgId}/applications/{applicationId}`: Application data, linking applicants to jobs within an organization.
- `/admins/{uid}`: A top-level collection to identify administrative users across the platform.

This structure ensures that all queries and operations are explicitly scoped to an organization, preventing any possibility of cross-tenant data leakage at the database level.

### 3.4. Secure Storage (Google Cloud Storage)

- Resumes are stored in a private Google Cloud Storage bucket.
- The storage path is scoped by organization and application: `resumes/{orgId}/{applicationId}/{filename}`.
- Strict Storage Security Rules deny all public access. Resumes are only accessible to authenticated admin users or via short-lived signed URLs generated by a secure Cloud Function.

## 4. Deployment and CI/CD

- The entire platform (Hosting and Functions) is deployed atomically using the Firebase CLI.
- The `firebase.json` configuration file defines the deployment targets, security rules, and indexes.
- The production build process is deterministic, installing dependencies from a frozen lockfile (`pnpm-lock.yaml`) to ensure reproducibility.
