# URAI-Jobs: Multi-Tenant Service Boundary Map

## 1. Core Principle: Tenant Isolation via Scoped Data Hierarchy

The fundamental principle of multi-tenancy in URAI-Jobs is the strict scoping of all tenant-specific data under a unique organization identifier (`orgId`). This is not an optional metadata tag but the root of the data hierarchy for all resources belonging to an organization. This architectural pattern makes cross-tenant data access impossible at the data layer.

**The root boundary is defined by the following Firestore path:**

`/orgs/{orgId}`

All data and resources belonging to a specific tenant (organization) *must* reside under this path.

## 2. Service & Data Boundaries

### 2.1. Cloud Firestore (Primary Data Store)

- **Boundary Enforcement**: Firestore Security Rules.
- **Mechanism**: All rules are predicated on matching the `orgId` from the request path with the user's authentication claims or other secure context. Any request attempting to access a path outside of the user's authorized `orgId` is denied by default.

**Data Silos:**

| Collection Path | Description | Access Boundary |
| :--- | :--- | :--- |
| `/orgs/{orgId}/jobs/{jobId}` | Private job data. | **Strictly** limited to authenticated users of `{orgId}`. |
| `/orgs/{orgId}/jobPublic/{jobId}` | Public job projection. | Read-only. The parent `orgId` is included for frontend filtering. |
| `/orgs/{orgId}/applicants/{applicantId}` | Applicant data for a tenant. | **Strictly** limited to authenticated users of `{orgId}`. |
| `/orgs/{orgId}/applications/{applicationId}` | Application data for a tenant. | **Strictly** limited to authenticated users of `{orgId}`. |
| `/admins/{uid}` | Platform-wide administrators. | **Global**. Not tenant-scoped. Access is highly restricted. |

**Example Security Rule Snippet:**
```json
match /orgs/{orgId}/{document=**} {
  allow read, write: if request.auth.token.orgId == orgId && isAdmin(request.auth.uid);
}
```

### 2.2. Cloud Storage for Firebase (PII / Resume Storage)

- **Boundary Enforcement**: Storage Security Rules.
- **Mechanism**: Similar to Firestore, all storage paths are required to be scoped by `orgId`. The rules validate that the `orgId` in the storage path matches the authenticated user's claims.

**Storage Silos:**

| Storage Path | Description | Access Boundary |
| :--- | :--- | :--- |
| `resumes/{orgId}/{applicationId}/{filename}` | Applicant resumes (PII). | **Strictly** private. No public read/write. Write is enabled via short-lived signed URLs generated by a Cloud Function that *knows* the `orgId`. Read is limited to authenticated admins of `{orgId}`. |

### 2.3. Cloud Functions (Backend Logic)

- **Boundary Enforcement**: Parameterized Firestore triggers and explicit `orgId` handling in callable functions.
- **Mechanism**: All database operations performed by Cloud Functions are explicitly scoped to the `orgId` derived from the trigger context or passed as a validated parameter.

**Function Boundaries:**

| Function | Trigger / Invocation | `orgId` Source | Boundary Enforcement |
| :--- | :--- | :--- | :--- |
| `onJobWrite` | `firestore.document('/orgs/{orgId}/jobs/{jobId}').onWrite` | `context.params.orgId` | The function *only* operates on the `jobPublic` collection *within the same `orgId`* path. Cross-tenant writes are architecturally impossible. |
| `onApplicationCreate` | `firestore.document('/orgs/{orgId}/applications/{appId}').onCreate` | `context.params.orgId` | All subsequent database writes (e.g., updating applicant activity, stats) are explicitly passed the `orgId` from the trigger context. |
| `adminSetApplicationStatus`| Callable | `context.auth.token.orgId` | The function verifies the caller is an admin for the *specific `orgId`* they are attempting to modify. |

## 4. Conclusion

The service boundaries are clear and enforced at multiple layers: the database schema, security rules (both Firestore and Storage), and the backend function logic. The `orgId` is the universal key that partitions the system, and no operation can cross this boundary. This makes the URAI-Jobs platform secure by design against cross-tenant data leakage.
