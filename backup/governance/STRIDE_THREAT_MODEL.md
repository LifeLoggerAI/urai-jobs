# URAI-Jobs: STRIDE Threat Model

This document provides a formal threat model for the URAI-Jobs platform based on the STRIDE framework. It identifies potential threats and documents the mitigations implemented within the system architecture.

## S - Spoofing

**Threat**: A malicious user impersonates a platform administrator to gain control over the system.

- **Mitigation**: Platform administrator status is determined by the existence of a document in the `/admins/{uid}` collection. Firestore Security Rules protect this collection, and access is granted only to authenticated users whose UID is present in this collection. Authentication is handled by Firebase Authentication, with recommended MFA enforcement.

**Threat**: A user from one tenant (e.g., `orgId: "acme"`) spoofs their identity to act as a user from another tenant (`orgId: "globex"`).

- **Mitigation**: This is the primary threat addressed by the multi-tenant architecture. All data is partitioned by `orgId`. Firebase Security Rules on both Firestore and Cloud Storage enforce that the `orgId` in the resource path (`/orgs/{orgId}/...`) *must* match the `orgId` claim in the user's JWT token (`request.auth.token.orgId`). This makes it architecturally impossible for a user from one tenant to spoof their identity and operate in another.

## T - Tampering

**Threat**: An applicant modifies or deletes another applicant's data.

- **Mitigation**: Firestore Security Rules grant applicants write access only for creating new documents in the `applications` and `waitlist` collections. They are explicitly denied read, update, or delete access to any documents other than their own, and even that is highly restricted.

**Threat**: A user from one tenant tampers with the job listings, applications, or applicant data of another tenant.

- **Mitigation**: The same `orgId`-based security rules that prevent spoofing also prevent tampering. All write operations (`create`, `update`, `delete`) are validated to ensure the user's `orgId` token claim matches the `orgId` in the document path.

## R - Repudiation

**Threat**: An administrator denies having performed a critical action, such as rejecting an applicant or deleting a job posting.

- **Mitigation**: All critical documents in Firestore (`jobs`, `applications`) have `createdAt` and `updatedAt` fields populated with `FieldValue.serverTimestamp()`. Furthermore, an `events` collection logs key activities (e.g., `application_submitted`, `status_changed`), providing a basic audit trail.

**Threat**: An applicant denies having submitted an application.

- **Mitigation**: The creation of an application document in `/orgs/{orgId}/applications/{applicationId}` with a `submittedAt` timestamp serves as a non-repudiable record of the submission event.

## I - Information Disclosure

**Threat**: An unauthorized party gains access to Personally Identifiable Information (PII), such as applicant resumes.

- **Mitigation**: This is a critical threat that has been mitigated through multiple layers. Resumes are stored in a private Google Cloud Storage bucket. The Storage Security Rules are configured to `allow read, write: if false;`, denying all access by default. Resumes can only be uploaded and downloaded via short-lived, single-purpose signed URLs generated by secure Cloud Functions (`createResumeUpload`, `getResume`). The `Applicant Data Flow Diagram` details this secure process.

**Threat**: Cross-tenant information disclosure, where a user from one organization sees data belonging to another.

- **Mitigation**: The strict, `orgId`-based data partitioning and corresponding security rules are the primary mitigation. This is the cornerstone of the platform's multi-tenant security, as detailed in the `Multi-Tenant Service Boundary Map`.

## D - Denial of Service (DoS)

**Threat**: A malicious actor floods the application submission endpoint, creating junk data and consuming resources.

- **Mitigation**: Firebase and the Google Cloud ecosystem provide a high degree of resilience to traditional network-layer DoS attacks. At the application layer, callable functions can be protected with Firebase App Check to prevent abuse from unauthorized clients. Further rate-limiting logic (e.g., by IP address or user ID) can be implemented within the functions if necessary.

**Threat**: Maliciously large file uploads are used to exhaust storage or bandwidth.

- **Mitigation**: The `createResumeUpload` Cloud Function validates file size and type *before* generating the signed URL. If the declared size is too large, the function returns an error, and no upload URL is generated, preventing the file from ever reaching Cloud Storage.

## E - Elevation of Privilege

**Threat**: A standard authenticated user elevates their privileges to become a tenant administrator or a platform administrator.

- **Mitigation**: Privilege is determined solely by server-side checks against the Firestore database (`/admins/{uid}` for platform admins, or a similar mechanism within an org for tenant admins). Firestore Security Rules are written to be stateless and cannot be influenced by client-side payloads. A user cannot "send" a parameter that grants them elevated privileges. The rules for modifying admin status are the most restrictive in the system, effectively preventing unauthorized elevation.
